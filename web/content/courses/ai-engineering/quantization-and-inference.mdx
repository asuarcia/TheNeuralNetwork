Training gets the headlines, but **inference** — actually running the model to serve users — is where the bills pile up and the latency hurts. A 70-billion-parameter model won't fit on a normal GPU at full precision, and a slow response loses users. This lesson is the engineering that makes big models practical.

## Why models are so big

A model's memory footprint is simply its parameter count times the bytes used to store each number. At full precision (FP32), each weight is 4 bytes; at half precision (FP16), 2 bytes.

  starter={`params = 7_000_000_000

`}
  expected="14.0"
  solution={`params = 7_000_000_000
bytes_per_param = 2
print(params * bytes_per_param / 1e9)`}
/>

14 GB just to *hold* the weights — before any actual computation. Scale to 70B and you need multiple high-end GPUs. That's the problem quantization solves.

## Quantization: fewer bits per weight

**Quantization** stores each weight using fewer bits — INT8 (1 byte), or even INT4 (half a byte) — instead of FP16/FP32. The model gets dramatically smaller and faster, at the cost of a little precision.

<CodeExercise
  id="inf-quant"
  prompt="FP16 uses 2 bytes/weight; INT4 uses 0.5. Print how many times smaller the INT4 model is."
  starter={`fp16_bytes = 2
int4_bytes = 0.5
# Print the size reduction factor (fp16 / int4)
`}
  expected="4.0"
  solution={`fp16_bytes = 2
int4_bytes = 0.5
print(fp16_bytes / int4_bytes)`}
/>

A 4× reduction can turn "needs a data-center GPU" into "runs on my laptop." The surprise of modern AI is how *little* quality you lose — a well-quantized INT4 model is often nearly indistinguishable from the full-precision original.

## Speeding up generation: the KV cache

Recall that a chat model is autoregressive — it generates one token at a time, re-reading the whole sequence each step. Naively, that means recomputing attention over *all* previous tokens for *every* new token. Wildly wasteful.

The **KV cache** fixes it: the Keys and Values computed for past tokens (from the self-attention lesson) are **stored and reused**, so each new token only computes attention against the cache instead of redoing everything. It's the single biggest reason generation is fast — at the cost of memory to hold the cache.

## Other inference levers

- **Batching** — process many users' requests together to keep the GPU saturated (throughput).
- **Smaller / distilled models** — a tuned 7B can beat a generic 70B for a specific task, far cheaper.
- **Streaming** — send tokens as they're generated so the user sees output immediately (better *perceived* latency).

<Callout type="tip">
The deployment mindset flips training on its head. In training you maximize learning; in inference you minimize cost and latency per request. Quantization, KV caching, and batching are the three biggest knobs — and together they decide whether your AI feature is profitable or a money pit.
</Callout>
