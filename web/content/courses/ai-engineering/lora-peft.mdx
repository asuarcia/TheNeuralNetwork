Full fine-tuning has a brutal cost: you update *every* weight (billions of them), need huge GPU memory to do it, and end up with a complete multi-gigabyte copy of the model per task. **Parameter-Efficient Fine-Tuning (PEFT)** sidesteps all of that — and **LoRA** is its star technique.

## The key insight

When you adapt a model to a task, the *change* to its weights turns out to be "low-rank" — it can be captured by a much smaller matrix than the full weight matrix. So instead of editing the giant weight matrix directly, LoRA does this:

- **Freeze** the original weights entirely — never touch them.
- **Inject** two small matrices alongside them, whose product approximates the needed change.
- Train **only those small matrices**.

A `d × d` weight matrix (millions of numbers) gets adapted by two skinny matrices: `d × r` and `r × d`, where the **rank** `r` is tiny (often 8 or 16).

## The savings are staggering

Compare that to full fine-tuning of the same layer and you see why LoRA took over.

62× fewer trainable parameters for *one* layer — and the gap widens with model size. In practice LoRA often trains **less than 1%** of a model's parameters while matching full fine-tuning's quality.

## Why this changed the game

Because the base model stays frozen, LoRA gives you superpowers:

- **Cheap training** — fits on a single consumer GPU instead of a cluster.
- **Tiny artifacts** — a LoRA "adapter" is a few megabytes, not gigabytes. You ship the adapter, not a new model.
- **Swappable & composable** — keep one base model in memory and hot-swap adapters for different tasks (a support-bot adapter, a coding adapter, a brand-voice adapter).
- **No catastrophic forgetting** — the original weights are untouched, so general ability is preserved.

<Callout type="tip">
The mental model: the big pretrained model is a shared, read-only foundation; each LoRA adapter is a small, swappable "skin" that specializes it. One base, many cheap personalities — that's the modern fine-tuning workflow.
</Callout>
