You can train a model. But a model that lives in a notebook helps no one. **MLOps** (machine-learning operations) is the discipline of turning a trained model into a reliable product — and it's where most real-world AI effort actually goes. This track, inspired by what cloud platforms like AWS SageMaker teach, walks the full lifecycle.

## The ML lifecycle is a loop, not a line

Production ML is a cycle that never really ends:

1. **Data** — collect, clean, and version the data.
2. **Train & experiment** — build models, track experiments, pick the best.
3. **Evaluate** — test on held-out data and against the current production model.
4. **Deploy** — package the model and serve it behind an API.
5. **Monitor** — watch quality, latency, cost, and drift in the wild.
6. **Retrain** — when the world shifts or data grows, loop back to step 1.

The first three you've practiced all curriculum. This track is about steps 4–6 — the part that separates a demo from a dependable system.

## Should this model ship?

A core lifecycle decision: only promote a new model if it beats the current one by a meaningful margin (not just noise). That guard prevents shipping a model that's no better — or worse.

<CodeExercise
  id="lifecycle-promote"
  prompt="Promote the new model only if its accuracy beats the current one by more than the margin. Print True or False."
  starter={`new_acc = 0.91
current_acc = 0.88
margin = 0.01
# Print True if new_acc beats current_acc by more than the margin
`}
  expected="True"
  solution={`new_acc = 0.91
current_acc = 0.88
margin = 0.01
print(new_acc > current_acc + margin)`}
/>

The new model is 3 points better, clearing the 1-point margin — ship it. At 0.885 it wouldn't have: too close to call.

<Callout type="tip">
Reproducibility is the foundation of all of this. If you can't recreate exactly which data + code + config produced a model, you can't debug it, audit it, or safely improve it. Version <em>everything</em> — data, code, and models — which is the subject of the last lesson.
</Callout>
