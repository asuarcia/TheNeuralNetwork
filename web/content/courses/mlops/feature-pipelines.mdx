Before a model trains or predicts, raw data must be turned into **features** — the clean numeric inputs the model expects. Doing this consistently, every time, in both training and production, is one of the hardest parts of real ML.

## The pipeline

A feature pipeline takes raw data and outputs model-ready features: clean missing values, encode categories, scale numbers, derive new fields. You built the ingredients in the Applied AI and Math tracks — here they become a *repeatable, automated* process that runs the same way every time.

## The silent killer: train/serve skew

Here's the bug that bites everyone. If features are computed **one way during training** and a **different way during serving**, the model sees inputs it was never trained on — and quietly makes bad predictions. This is **train/serve skew**, and because nothing crashes, it's brutal to catch.

The classic cause: **normalization stats**. If you scale a feature using its mean and standard deviation, you must use the **training** mean/std at serving time — not recompute them from live data. Reusing the saved training stats keeps train and serve identical.

train_std = 2

`}

  solution={`x = 8
train_mean = 4
train_std = 2
print((x - train_mean) / train_std)`}
/>

## Feature stores

The professional fix is a **feature store** — a central system that computes and stores features once, so both training and serving read the *exact same* definitions. It also enables reuse (teams share features) and point-in-time correctness (no accidentally using future data). Cloud platforms (like SageMaker Feature Store) provide these as managed services.

<Callout type="tip">
The golden rule: <strong>compute a feature one way, in one place.</strong> Whether via a shared library or a feature store, eliminating the chance for training and serving to diverge prevents a whole category of silent, painful bugs.
</Callout>
