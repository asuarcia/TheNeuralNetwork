Here's the uncomfortable truth of production ML: your model was trained on the past, but it predicts the future, and the world keeps changing. Performance silently erodes. **Monitoring** is how you catch that — ideally before it costs you.

## What to monitor

Two layers:

- **Operational health** — latency, throughput, error rate, cost. The same things any service watches.
- **Model quality** — accuracy/precision over time, prediction distributions, and **drift**. This is what's unique (and easy to forget) about ML systems.

Start with the basics — an error rate is just failures over total requests.

## Drift: when the world moves

**Drift** is the core ML-specific failure. Two flavors:

- **Data drift** — the *inputs* change. A model trained on last year's shopping behavior sees this year's different habits.
- **Concept drift** — the *relationship* changes. What counted as fraud last year isn't what counts now.

Either way, the model gets relatively worse without any code changing. You detect it by comparing a live quality metric against your launch **baseline** and alerting when it slips too far.

<CodeExercise
  id="mon-drift"
  prompt="Baseline accuracy was 0.92; current is 0.80. Alert if it dropped more than 0.05. Print 'alert' or 'ok'."
  starter={`baseline = 0.92
current = 0.80
threshold = 0.05
# Print "alert" if the drop exceeds the threshold, else "ok"
`}
  expected="alert"
  solution={`baseline = 0.92
current = 0.80
threshold = 0.05
print("alert" if baseline - current > threshold else "ok")`}
/>

A 12-point drop trips the alarm — you find out from monitoring, not from a furious customer.

## The catch: labels arrive late

Measuring accuracy live needs the *true* answer — which often comes much later (did this flagged transaction turn out fraudulent? did the user actually churn?). So teams also watch **proxy signals** that are available immediately:

- Shifts in the **input distribution** (statistical drift tests).
- Shifts in the model's **output distribution** (suddenly predicting "fraud" 3× as often).
- **Business metrics** (click-through, conversion) tied to the model.

When monitoring fires, the response is usually: investigate, then **retrain** on fresh data — closing the lifecycle loop.

<Callout type="tip">
Set up monitoring <em>before</em> you launch, not after the first incident. The cheapest moment to catch degradation is the moment it begins. A model with no monitoring is a model you're flying blind.
</Callout>
