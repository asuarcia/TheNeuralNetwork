**Serving** is making your model available to do inference on demand. The trained model is just a file of weights; serving wraps it so applications (and users) can send inputs and get predictions back — at the speed and scale they need.

## Batch vs. real-time

The first decision: how do predictions get used?

- **Batch (offline)** — run the model on a big pile of data on a schedule (e.g., score every customer's churn risk nightly). Simple, cheap, high-throughput. Fine when predictions aren't needed instantly.
- **Real-time (online)** — wrap the model in an API that responds to each request in milliseconds (e.g., fraud check at checkout). Harder: you must hit a **latency** target under live load.

## Latency and throughput

Two numbers define a real-time service:

- **Latency** — how long one request takes. Usually measured at the **tail** (p95/p99), because the slowest requests are what users feel — not the average.
- **Throughput** — how many requests per second you can handle. Divide total requests by the time taken.

You also need to know when you're **breaching your SLA** (service-level agreement) — the latency budget you promised. Counting requests over budget is the simplest health check.

## Scaling and cost

Under load, one server isn't enough. The usual levers:

- **Horizontal scaling** — run many copies behind a load balancer; **autoscale** the count with traffic.
- **Batching requests** — group incoming requests so the GPU stays busy (great for throughput, trades a little latency).
- **The inference tricks from AI Engineering** — quantization and KV caching make each prediction cheaper and faster.

Cloud services (SageMaker endpoints, Bedrock for hosted LLMs) handle much of this for you — but the trade-offs are the same whoever runs the servers.

<Callout type="tip">
There's no single "fast enough." Define your targets up front — "p95 latency under 200ms at 1,000 requests/second for under $X/month" — then engineer to them. Vague goals like "make it fast" can't be met or measured.
</Callout>
